# Webinar-Prep

Операционная готовность эфира к прямому включению: спикер — контент — техника — прогон — день эфира.

Это не про «о чём говорить» (контент — `methodist`) и не про «как зазвать» (анонсы — `announce-writer`). Это про то, чтобы в момент «мы в эфире» ничего не отвалилось: спикер на связи, звук есть, демонстрация работает, ссылки живые, у ведущего на руках протокол на случай сбоя.

---

## Когда использовать скилл

Активировать, когда задача связана с:
- Подготовкой к прямому эфиру за 1–2 недели до даты
- Сбором чек-листа готовности спикера (что прислать, что прорепетировать)
- Технической проверкой площадки (платформа трансляции, звук, демонстрация экрана, запись)
- Прогоном / репетицией эфира
- Протоколом дня эфира — что делать по минутам и что делать, если что-то ломается

**Не использовать для:** написания анонсов и прогревов (`announce-writer`), структуры слайдов и тезисов (`methodist`), концепции и механики запуска (`launch-strategist`), ТЗ на монтаж записи (`video-edit-assistant`).

## Источники правды

Перед сборкой открыть:
- `../../knowledge-base/speakers.md` — регалии спикера, как любит, чтобы его представляли, стиль подачи, технические привычки.
- `../../knowledge-base/zerocoder-facts.md` — даты, ссылки, продукты, оффер, который прозвучит на эфире.
- `../../knowledge-base/audience.md` — кто в зале, под какой уровень настраивать демонстрации и темп.

Если факта (дата, ссылка, платформа, регалии) нет — спросить Повелительницу, не выдумывать.

## Алгоритм работы

### Шаг 1. Снять параметры эфира

Зафиксировать до сборки:
- Формат: вебинар разговорный / бизнес-разбор / интенсив-модуль / инфо-эфир (напр. совместный с IU)
- Дата, время, часовой пояс, плановая длительность
- Платформа трансляции (и резервная, если есть)
- Спикер(ы) — один или несколько, кто ведущий, кто модератор
- Будет ли демонстрация экрана / инструментов (демо ИИ-сервисов — зона риска: даты, личные данные, лишние вкладки)
- Идёт ли запись (и куда уходит — для последующего монтажа)
- Что в конце: оффер, регистрация на следующий шаг, опрос, Q&A

Жидкие вводные — 2–3 уточняющих вопроса, дальше не додумывать.

### Шаг 2. Собрать четыре блока готовности

По каждому блоку — статус (готово / в работе / риск) и ответственный.

1. **Спикер** — что прислать, что прорепетировать, как представят.
2. **Контент** — финальные слайды, тайминг по частям, демонстрации проверены.
3. **Техника** — площадка, звук, видео, демонстрация экрана, запись, ссылки.
4. **Прогон** — репетиция со спикером, проверка переходов и тайминга.

### Шаг 3. Написать протокол дня эфира

По минутам: за сколько до старта кто где, как стартуем, кто следит за чатом, как закрываем. Отдельно — **план на сбой** (см. ниже).

### Шаг 4. Оформить и передать

Сохранить по формуле из CLAUDE.md: `[дата]_[тема]_prep.md` (напр. `2026-06-05_webinar-openclaw_prep.md`).
Передать Повелительнице. Спикеру отправляет она — скилл готовит черновик памятки, сам никуда не шлёт.

## Блок 1. Спикер — чек-лист

- [ ] Тема и тезисы согласованы со спикером (не за день — минимум за 3–4 дня)
- [ ] Спикер знает свой тайминг (сколько минут на блок, когда Q&A)
- [ ] Финальные слайды у спикера на руках и проверены им
- [ ] Согласовано представление спикера (точные регалии из `speakers.md`, как он любит)
- [ ] Спикер знает оффер/CTA и когда его озвучить
- [ ] Технические привычки учтены (с какого устройства, демонстрирует ли свой экран)
- [ ] У спикера есть прямая связь с ведущим на время эфира (резервный канал — телефон/мессенджер на случай вылета)
- [ ] Памятка спикера отправлена (что подготовить, во сколько подключиться, что иметь под рукой)

## Блок 2. Контент — чек-лист

- [ ] Слайды финальные, без рабочих пометок и старых дат на экране
- [ ] Тайминг расписан по частям, сумма ≤ плановой длительности с запасом на Q&A
- [ ] Все демонстрации (инструменты, ИИ-сервисы) проверены вживую в день прогона
- [ ] На экранах демо нет лишнего: чужих вкладок, закладок, уведомлений, личных данных, старых дат
- [ ] Ссылки, которые показываем/кидаем в чат, открываются и ведут куда нужно
- [ ] Заготовлены ответы на 3–5 частых возражений/вопросов
- [ ] Финальный экран с CTA готов (куда вести зрителя после)

## Блок 3. Техника — чек-лист

- [ ] Площадка трансляции проверена, доступ у спикера и ведущего есть
- [ ] Резервный сценарий, если основная платформа падает (вторая ссылка / переход в др. сервис)
- [ ] Звук спикера: микрофон, отсутствие эха, уровень громкости
- [ ] Видео: свет, фон, камера на уровне глаз
- [ ] Демонстрация экрана работает на той машине, с которой будет спикер
- [ ] Запись включается и пишет в нужном качестве (для последующего монтажа)
- [ ] Чат/вопросы: кто модерирует, куда складываем вопросы для Q&A
- [ ] Интернет: проводное соединение или проверенный канал, резерв (мобильный) под рукой

## Блок 4. Прогон — чек-лист

- [ ] Репетиция со спикером проведена (минимум за 1–2 дня)
- [ ] Проверены переходы между блоками и кто говорит после кого
- [ ] Замерен реальный тайминг — укладываемся ли
- [ ] Проверены все демонстрации именно в боевом окружении
- [ ] Спикер знает сигналы ведущего (время, переход к Q&A, завершение)

## Протокол дня эфира

| Время | Кто | Действие |
|---|---|---|
| −60 мин | Ведущий | Поднял площадку, проверил запись, открыл чат |
| −30 мин | Спикер + ведущий | Созвон, проверка звука/экрана/демо |
| −15 мин | Ведущий | Открыл «комнату ожидания», проверил ссылку в анонсе |
| −5 мин | Все | Тишина в эфире, финальная проверка записи |
| 0 | Ведущий | Старт: приветствие, регламент, представление спикера |
| ход эфира | Модератор | Следит за чатом, собирает вопросы, помечает тайминг |
| финал | Ведущий | CTA, следующий шаг, благодарность, мягкое завершение |
| +5 мин | Ведущий | Остановил запись, сохранил, проверил целостность файла |

### План на сбой (обязательный блок)

Заранее ответить на «а если»:
- **Спикер вылетел из эфира** → ведущий держит паузу контентом/вопросами, спикер заходит по резервной ссылке; связь с ним по телефону/мессенджеру.
- **Падает площадка трансляции** → заранее заготовлена вторая ссылка; её быстро кидаем в чат и анонс-бот.
- **Не работает демонстрация** → есть скриншоты/запись демо как запасной вариант, показываем их.
- **Нет звука у спикера** → переключаемся на резервное устройство / телефон, ведущий комментирует, пока чинят.
- **Запись не пишется** → второй источник записи (локальная запись у спикера / OBS), проверено на прогоне.

## Чек-лист перед сдачей Повелительнице

- [ ] Все четыре блока готовности заполнены, по каждому пункту проставлен статус и ответственный
- [ ] Даты, время, ссылки, регалии спикера — из `knowledge-base`, не выдуманы
- [ ] Протокол дня эфира расписан по времени
- [ ] План на сбой закрывает минимум 5 сценариев
- [ ] Памятка спикера готова отдельным черновиком (Ольга отправит сама)
- [ ] Файл назван по формуле `[дата]_[тема]_prep.md`
- [ ] Ничего не отправлено спикеру/во внешние сервисы напрямую — только черновики на проверку

## Опорные правила

- **Готовность, а не контент.** Не лезть в тезисы и формулировки слайдов — это `methodist`. Задача скилла — чтобы эфир физически состоялся.
- **Каждый риск — с планом B.** Любой пункт техники, у которого есть шанс отвалиться, сопровождается резервным сценарием.
- **Демо — зона риска.** На демонстрации инструментов чаще всего утекают личные данные и мелькают старые даты — этот пункт проверять отдельно и въедливо (стыкуется с `video-edit-assistant` на этапе монтажа записи).
- **Никаких внешних отправок.** Памятки и ссылки уходят Повелительнице, она рассылает спикерам сама.
- **Регистр эфира учитывать.** Инфо-эфир с партнёром (напр. IU) — без продающего нажима в финале; протокол и CTA подстраивать под академический регистр.
